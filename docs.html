How TRIAD Works

TRIAD is a personal control room for AI — one dashboard that routes work across Claude, ChatGPT, Gemini, and anything else you plug into it, instead of juggling five browser tabs. This page walks through everything under the hood: how a prompt turns into a plan, how a plan turns into real running tasks, how a model asks you a question mid-task, and how the whole thing stays in sync in real time. Written the way we'd want it explained to us.

What TRIAD Is

At its simplest, TRIAD does one thing: you describe what you need, and it figures out which connected AI service is best suited for each piece of that request, runs it for real, and shows you the result — live, without needing to refresh. No frameworks, no build step. It's plain HTML, CSS, and JavaScript, matching the rest of this site's approach: open the file, everything's right there to read.

The name comes from the original idea: three models working together instead of three separate conversations. It's grown since then — TRIAD now happily juggles more than three services, including free ones, paid ones, image/video/audio generators, and anything else with an API that speaks a compatible language. The name stuck because the idea didn't change, just the roster.

How It's Built

TRIAD is three separate pieces that each do exactly one job, and nothing more:

Your browser — the dashboard itself, a single HTML file
↕ Firebase Auth → anonymous sign-in, just enough identity to scope your data
↕ Firestore → tasks, logs, and requests — synced live via onSnapshot
↕ Cloudflare Worker → the only thing that ever touches API keys or calls a model
↕ Workers KV → where keys actually live, server-side, never in the browser

The reason it's split this way instead of one big app: a browser tab is not a safe place to keep an API key. Anyone with dev tools open can read anything sitting in frontend JavaScript. So the Worker exists as a wall — the dashboard tells it "run this prompt on Claude," the Worker looks up the actual key (which the browser has never seen and never will again after the moment it was typed in), makes the real call, and hands back a result. The key itself never crosses back over that wall.

Firestore is a separate concern entirely — it's not for secrets, it's for everything that needs to feel alive: task progress ticking up, a request appearing the moment a model asks for something, a log entry landing the second a task finishes. Firestore's security rules pin every document to the signed-in session that created it, so even though the database is technically reachable from any browser on the internet, nothing in it is readable by anyone but you.

Why Cloudflare instead of Firebase Functions

The obvious choice for "run backend code near Firebase" is Firebase Cloud Functions. We tried that first. It turns out Cloud Functions require Google's paid Blaze plan the moment you need to make an outbound network call — even a single request to Anthropic's API counts as "outbound," so the free Spark plan can't do it at all. Cloudflare Workers, on the other hand, offer a genuinely free tier (100,000 requests a day, no credit card) that has no such restriction. So Firestore and Auth stayed on Firebase, and the actual "talk to an API" logic moved to a Cloudflare Worker instead. Two providers, each doing what they're actually free at.

What's Live vs. Limited

Not everything in TRIAD works the same way under the hood, and being upfront about that matters more than pretending everything's uniformly "real":

Overview Page: Rate Limit Gauges

This is the landing page once you're inside the dashboard. Each gauge shows a connected model's remaining rate-limit headroom as a percentage, pulled from the actual response headers a provider sends back with every API call. Anthropic and OpenAI both return ratelimit-remaining-style headers on every single response; Groq, since its API mirrors OpenAI's shape, does too.

A gauge only appears once two things are true: the service has a saved key, and it's actually capable of reporting usage. Gemini, Mistral, and Workers AI are usable everywhere else in the app but never show a gauge here, because there's genuinely nothing to show — those providers don't send the headers this feature depends on. Rather than fake a number, TRIAD just doesn't draw the gauge.

Each gauge carries a small badge — LIVE once real numbers have been pulled at least once, MOCK before that's happened. Hitting Refresh Usage in the top-right makes one genuine, tiny API call to each trackable service purely to read its headers, then stores the result so it survives a refresh. It's a button and not an automatic poller on purpose — every refresh is a real request against your actual quota, however small.

Tool Credit Cards

Below the reasoning-model gauges sits a second row for capability-specific tools — currently Meshy (3D generation) and Suno (music generation). These work on a credit system rather than a rolling rate-limit window, so they get their own compact card showing credits used against the total, once connected.

The New Prompt Flow

Click + New Prompt from Overview and a plain textarea appears — describe what you need in ordinary language, no special syntax, no need to say which model should do what. Hit Break It Down and TRIAD reads the request and proposes a plan: a short list of steps, each one assigned to whichever connected service fits, with a one-line reason why.

That plan is a proposal, not a commitment. Every step has a dropdown — override any assignment before anything runs. Once it looks right, Confirm & Start is what actually kicks things off: real tasks get created, and the app jumps you straight to the Tasks page to watch them go.

How Decomposition Works

The splitting logic looks for two kinds of signal in what you typed. First, capability keywords — words like "3D," "mesh," "model" (in the sculpting sense), or "song," "music," "soundtrack" — route straight to whichever connected tool specializes in that (Meshy or Suno). A reasoning model, however capable, genuinely can't output a 3D mesh or an audio file, so this routing never falls back to one; if the right tool isn't connected, the plan says so plainly instead of quietly assigning something that can't actually do the job.

Everything else — the bulk of most requests — splits across a default three-stage pattern: scope & structure first (turning a loose idea into something concrete), core build second (the actual implementation), review & catch edge cases third (a second model taking a fresh pass over what the first one built). Three different services touch the request by default, each doing the part it's naturally suited for.

On the word "decompose": right now this splitting logic is deterministic pattern-matching, not a model call in itself — fast, free, and predictable, but it doesn't understand your request the way a model would. A more capable version would send the raw prompt to a model and ask it to propose the plan directly; the plan format is already shaped to support that swap without touching anything downstream.

Routing & Fallback

Settings → Routing controls two things. Default model lets you skip the three-stage split entirely and force every reasoning step in every plan onto one service — useful if you've decided, say, Groq is your daily driver and you don't need three different opinions on every request. Fallback order is a reorderable priority list: if a plan step lands on a model that isn't actually connected, TRIAD walks down that list and reassigns to the first one that is, rather than proposing a dead end.

Capability-locked tools sit outside this system on purpose — Meshy stays Meshy, Suno stays Suno, regardless of default model or fallback order, since nothing else can substitute for what they do.

How a Task Actually Runs

Confirming a plan doesn't run everything at once. Steps execute one at a time, in the order shown in the plan review — deliberately simple, since it matches the "runs after step N" dependency labels already shown before you confirm, and avoids the complexity of figuring out which steps could safely run in parallel. Each step's flow looks like this:

  1. A task document is created in Firestore with status queued — the real-time listener picks it up and a card appears on the Tasks page immediately.
  2. Status flips to running, and the actual prompt gets sent to the Worker, which sends it to the real provider.
  3. One of three things happens: the model finishes normally (status becomes complete, full output saved to the task), it fails for a real reason like a bad key or a rate limit (status becomes error, the actual error message saved), or it signals it needs your input before it can continue (status becomes blocked — see the clarification section next).
  4. A log entry gets written either way, with a short preview of what happened.

How a Model Asks for Help

This is the piece that makes TRIAD feel less like a fire-and-forget tool and more like something you're actually working alongside. There's no API-level signal any provider sends for "I need clarification" — that had to be built as a convention. Every reasoning-model prompt gets wrapped, behind the scenes, with an instruction: if the task can't be completed without something only you can provide — a decision, account access, a missing piece of context — respond with a specific structured marker instead of guessing.

The Worker checks every response for an exact {"needs_input": true, "question": "..."} payload. That check is deliberately narrow — it only fires on that precise shape, so a task that legitimately asks for JSON as its actual deliverable doesn't get misread as a request for help.

Worth knowing: this is a convention the model is asked to follow, not something enforced by the API itself. Strong models tend to follow it reliably. Smaller or free models sometimes don't — they might guess instead of asking, or ask in the wrong format and have it go unnoticed. That's a real, structural limitation of doing this without native provider support, not a bug to be patched away.

The Requests Page

When a task gets blocked this way, a new document lands in the requests collection — tagged with which model asked, the exact question, and the task it's tied to. It shows up on the Requests page immediately, in real time, no refresh needed.

Click a card to expand it. If it's still Open, there's a text box to type your answer — account details, a decision, whatever it needs — and a Send Response button. That moves it to Responded, and a second button appears: Mark Complete & Notify.

Resuming a Blocked Task

This is the step that closes the loop rather than leaving it half-finished. Clicking Mark Complete & Notify doesn't just file the request away — if it's tied to a task, that task actually resumes. The model gets called again, fed back its own original question plus your answer, with instructions to continue and finish the original task using that new information. The task's status moves from blocked back through running to (hopefully) complete, same as any other task.

If the model needs a second round of clarification after that, the same mechanism creates another linked request rather than looping silently — you'll always see exactly what's being asked and when.

Reading the Tasks Page

Every task shows its title, assigned model, a progress bar, and a status badge — Queued, Running…, Complete, Failed, or Awaiting your input. Failed tasks show the real error message underneath, not a generic "something went wrong" — if a key's missing, it says so; if a provider returned a specific error, that's what you see.

Completed tasks with real output get a View output link — expands to the full response in a scrollable panel, with a one-click Copy button.

Plan Grouping & Combined View

Every step generated from a single New Prompt run shares a planId, so instead of scattering three or four related steps as separate, disconnected cards, they group under one header showing your original request and a live "N/M steps complete" counter. Expand the group to see each individual step, or hit View combined output to see every step's response stacked together with clear labels — the scope step, the build step, and the review step, all in one scroll, which is exactly what you want when the pieces genuinely belong together (like a homepage build where the "review" step is meant to be read alongside the "build" step it's reviewing).

Single-step tasks, and anything from before this feature existed, render exactly as a plain card with no group wrapper — there's nothing to combine, so nothing pretends there is.

Code Extraction & Downloads

Any time a task's output contains fenced code blocks — the standard ```language ... ``` markdown convention every model already uses when writing code — TRIAD pulls them out automatically. Each block gets a guessed filename based on its declared language (a ```python block becomes file-1.py, ```html becomes file-1.html, and so on) and a real Download button that saves it as an actual file through the browser, no copy-paste required.

For a grouped multi-step plan, there's also a Download all code button that gathers every code block across every step of that plan and downloads them all in sequence — useful when, say, the build step wrote the HTML and the review step suggested a separate CSS fix, and you want both files at once.

Log Check

A running, chronological record of everything that's happened — every completed task, every failure, every time a model asked for clarification — with date, time, and which service was involved. It's the page to check when you want to know what actually took place across a session without digging through individual task cards.

Settings, In Full

Connected Services

Every service TRIAD knows about lives here — built-ins (Claude, ChatGPT, Gemini, Groq, Mistral, Workers AI, Meshy, Suno) plus anything added manually. Each row shows its capability, a masked key field once connected, and the actual controls:

  • Save & Connect — sends a pasted key straight to the Worker, which stores it in KV. The browser never holds onto the real value after this point; the field shows a masked placeholder from then on.
  • Change — clears the field for re-entry. The old key stays valid server-side until a new one is actually saved over it.
  • Remove — genuinely deletes the key from the backend, not just a local UI reset. Confirms first, since it means re-pasting the key to reconnect later.
  • Hide (built-ins only) — tucks a service out of the main list without touching its connection state at all. It's still fully usable everywhere else in the app; it just stops cluttering Settings if you don't use it. A small "N hidden services" toggle at the bottom brings any of them back into view.
  • Delete (custom services only) — fully removes it from TRIAD's registry, not just a disconnect. Built-ins can't be deleted this way since the Worker has specific code written for each of them; anything you add yourself can be.

Routing

Covered above in detail — default model and fallback order, both of which feed directly into how decompose() assigns steps.

Adding a Custom Service

Settings → Connected Services → + Add Another Service opens a small form instead of a string of pop-ups. Two categories:

Tool

Task-specific generators — image, video, voice, anything outside general reasoning. No API endpoint required at creation; TRIAD just tracks that it exists and lets you route to it manually.

Reasoning model

A general-purpose model that can take part in the normal scope/build/review split. Requires an API endpoint (base URL) — must accept the same request shape as OpenAI's /chat/completions endpoint, which is true of most providers today (OpenRouter, Together, Fireworks, DeepSeek, and plenty of others). Once added, it shows up in every model dropdown throughout the app exactly like a built-in.

A key can be added at creation time or skipped and added later from the same Settings row as everything else.

Security Model

  • API keys live only in Cloudflare Workers KV. The browser sends a key exactly once, at save time, and can never read it back — the UI shows a masked placeholder, not the real value, from that point on.
  • The Worker is gated by a single shared passphrase sent as a header on every request. That's intentionally simple — TRIAD is a single-user tool, not a multi-tenant service, so there's no need for individual accounts on the Worker side.
  • Firestore data — tasks, logs, requests — is scoped per-uid through security rules. Readable and writable only by the session that created it, enforced server-side, not just hidden in the UI.
  • Firebase's client-side apiKey, visible in the page source, is not a secret in the way it sounds — Google's own documentation confirms this value is meant to be public; it identifies the project, it doesn't grant access on its own. The real access boundary is the security rules and Auth, not this string.

Firestore Schema

Four collections, each scoped to uid:

tasks/{taskId}
  title, model, pct, status, output, error,
  planId, planSummary, planSteps, stepIndex,
  uid, createdAt

logs/{logId}
  model, text, uid, timestamp

requests/{requestId}
  model, text, status, response, respondedAt,
  askedAt, taskId, originalPrompt, uid, createdAt

settings/{uid}
  (routing preferences — not yet persisted here, see Known Limitations)

Known Limitations

Nothing below is hidden or papered over — this is the honest list of what doesn't fully work yet:

  • Routing settings and hidden services don't persist. Default model, fallback order, and which built-ins are hidden all live in browser memory only right now — they reset on every refresh. A settings/{uid} Firestore document exists in the rules but isn't being written to yet.
  • Gemini, Mistral, and Workers AI can't show live rate-limit gauges — a real limitation of those providers, not something fixable in TRIAD's code.
  • Meshy and Suno integration is unverified against current live traffic — written against typical REST conventions, worth double-checking against current docs before depending on it.
  • Tasks run sequentially, even when steps don't strictly depend on each other. Simple and predictable, but not the fastest possible execution for plans with independent steps.
  • Model provider strings drift. Providers deprecate specific model IDs on their own schedule (this has already happened twice during development — once with Cloudflare's Workers AI catalog, once with Gemini's 2.0 series). If a service suddenly starts failing with a "deprecated" or unexpected 429/404 error despite a valid key, the model string in the Worker is the first thing worth checking against that provider's current docs.
  • The clarification loop is a convention, not a guarantee — covered in detail above, worth repeating here since it's the single biggest source of "why didn't this create a Request" confusion.

Running Your Own Copy

TRIAD is static — no build step, no server required for the frontend itself. It's three HTML files (the landing page, the dashboard, this documentation) that can be opened directly or served from anywhere that hosts static files, GitHub Pages included.

Standing up a working copy means three things existing at once: a Firebase project with Firestore and Anonymous Auth enabled (free, no billing needed), a deployed Cloudflare Worker holding the API keys and doing the actual provider calls (also free, no card needed), and the dashboard's own config pointing at both. The Worker replaces what would otherwise need to be a paid Cloud Functions setup — see the architecture section above for why that split exists.

Once both pieces are live and their connection details are in place, everything else — routing, adding services, running prompts — works entirely inside the dashboard itself. No further backend changes needed for day-to-day use.
