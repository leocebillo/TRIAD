How TRIAD Works

One console for Claude, ChatGPT, Gemini, and anything else you connect — what's real, what's still a demo, and how to run your own copy.

What TRIAD Is

TRIAD is a personal dashboard for routing work across multiple AI services instead of juggling separate tabs. You type one request, it gets split into pieces matched to whichever connected service fits each piece best, and you watch real rate-limit usage, task progress, and a running log — all from one place.

It's built with no frameworks — plain HTML/CSS/JS on the frontend, matching the rest of this site's structure. There's no build step; the dashboard is a single file you can open directly or host anywhere that serves static files.

Architecture

Three pieces, each doing one job:

Browser (triad-dashboard.html)
  ├─ Firebase Auth → anonymous sign-in, gives you a stable identity
  ├─ Firestore → tasks / logs / requests, synced in real time
  └─ Cloudflare Worker → holds API keys, calls each provider
      └─ Workers KV → where keys actually live, never in the browser

The split matters for one reason: your API keys never touch the browser after the moment you paste them. They go straight to the Worker, which stores them server-side in KV and is the only thing that ever makes the real calls to Claude/ChatGPT/Gemini/etc. The dashboard just asks the Worker to run something and gets a result back.

Firestore is separate from that — it's not for secrets, it's for everything the UI needs to sync live: task progress, the activity log, and requests. Firestore's security rules scope every document to your own signed-in uid, so even though the database is technically reachable from any browser, nothing readable belongs to anyone but you.

Live vs. Limited

Not everything shown in TRIAD is equally "real" — here's the honest breakdown so nothing surprises you once you're relying on it:

FeatureStatusNotes
Tasks, logs, requestsReal-timeStored in Firestore, synced via onSnapshot — survives refresh, updates live across tabs.
New Prompt executionRealConfirming a plan actually calls the Worker, which actually calls the model.
Claude / ChatGPT / Groq usage gaugesReal (on refresh)Pulled from actual rate-limit response headers. Click "Refresh Usage" to update — it's a real API call, so it's manual, not polled.
Gemini / Mistral / Workers AI gaugesNot trackableThese providers don't expose per-request rate-limit headers at all. Fully usable, just not measurable the same way.
Automatic model-initiated requestsNot built yetNothing currently detects "this model needs clarification" and creates a Request automatically — that needs the model call to signal it explicitly, which isn't wired up.
Meshy / Suno callsUnverifiedWritten against each provider's typical REST shape but not confirmed against live docs — check before relying on it.

The Five Pages

Overview

Rate-limit gauges for whatever's connected and capable of showing real usage, plus a recent-activity feed. Nothing shows here until you've connected at least one service with header-based tracking (Claude, ChatGPT, or Groq).

Log Check

Every completed (or failed) task, with date, time, and which service handled it. This is the running record — mostly useful for looking back at what actually happened across a session.

Requests

Where you'd respond to something a model flagged mid-task — approve a decision, hand over account details, whatever it needs. Right now these are only created by you or seeded as demo data, since automatic model-initiated requests aren't built (see Live vs. Limited).

Tasks

Live status for anything running or finished — queued, running, complete, or failed, with the actual error message shown if something broke.

Settings

Connect or remove services, set routing preferences, and manage what's visible. Covered in detail below.

New Prompt & Routing

Click + New Prompt on Overview, describe what you need in plain language, and hit Break It Down. TRIAD splits it into steps and proposes a service for each one, with a short reason why — you can override any assignment before confirming.

Routing works two ways, set in Settings → Routing:

  • Auto (best fit) — the default. Capability keywords (3D, music, etc.) route to Meshy/Suno; everything else splits across a scope → build → review pattern.
  • A specific default model — overrides the split entirely and sends reasoning steps to whichever model you've picked, falling back through your configured Fallback order if that model isn't actually connected.

Capability-specific tools (Meshy, Suno) never silently fall back to a text model — there's no reasonable substitute for "generate a 3D asset" among reasoning models, so a disconnected tool just gets flagged instead of misrouted.

Handling Requests

Click a request card to expand it. If it's Open, type your response and hit Send Response — that moves it to Responded. From there, Mark Complete & Notify closes the loop. Every state change writes straight to Firestore, so it's reflected instantly if you have the dashboard open in another tab.

Adding a Service

Settings → Connected Services → + Add Another Service. Two categories:

  • Tool — image/video/audio generation, anything task-specific. No endpoint required at creation.
  • Reasoning model — needs an API endpoint (base URL). Must accept the same request shape as OpenAI's /chat/completions — true for most providers today (OpenRouter, Together, Fireworks, DeepSeek, and others).

Built-in services (Claude, ChatGPT, Gemini, Groq, Mistral, Workers AI, Meshy, Suno) can be Hidden from view if you don't use them, but not deleted — the Worker has specific code for each. Anything you add yourself can be fully Deleted.

Firebase Setup

Firestore and Auth run on Firebase's free Spark plan — no billing required for this part.

  1. Create a project at console.firebase.google.com
  2. Build → Firestore Database → Create database (production mode)
  3. Build → Authentication → Sign-in method → Anonymous → Enable
  4. Project settings → General → Your apps → Web (</>) — register an app, copy the config object
  5. Paste that config into triad-dashboard.html, where it says firebaseConfig
Deploy the security rules too. firestore.rules is what actually enforces that your data stays yours — deploy it with firebase deploy --only firestore:rules, or nothing will be readable/writable at all.

Cloudflare Worker Setup

The Worker is what holds your API keys and makes real calls to each provider — it deliberately replaces Firebase Cloud Functions, since those require the paid Blaze plan just to make outbound requests. Cloudflare's free tier (100,000 requests/day) needs no card at all.

npm install -g wrangler
wrangler login          # or use an API token if OAuth can't reach your browser
wrangler kv namespace create TRIAD_KEYS
# paste the id it prints into wrangler.toml
wrangler secret put DASHBOARD_TOKEN
wrangler deploy

Copy the resulting *.workers.dev URL and your DASHBOARD_TOKEN into triad-dashboard.html where it says WORKER_URL and DASHBOARD_TOKEN.

Cloud Shell / headless environments: wrangler login tries to open a browser callback that often can't be reached from a cloud terminal. Use an API token instead: create one at dash.cloudflare.com/profile/api-tokens with Workers Scripts + Workers KV Storage edit permissions, then export CLOUDFLARE_API_TOKEN=your_token before running any wrangler command.

Deploying to GitHub Pages

The dashboard is static — no build step, no server needed for the frontend itself. GitHub Pages serves it as-is.

  1. Push index.html, triad-dashboard.html, and this docs.html to a GitHub repo
  2. Repo Settings → Pages → Source → deploy from your main branch
  3. Once live, your dashboard URL becomes the one place you actually use — the Firebase config and Worker URL are already baked into the file, nothing else to configure
Before pushing: double-check triad-dashboard.html doesn't contain a real, currently-valid API key pasted directly into the file anywhere. Keys belong in Settings (which sends them to the Worker/KV), never hardcoded into the HTML itself — GitHub Pages repos are public unless you're on a paid plan with private Pages.

Security Model

  • API keys live only in Cloudflare Workers KV. The browser sends a key once, on save, and never sees it again — renderServices() shows a masked placeholder, not the real value.
  • The Worker is gated by a single shared passphrase (DASHBOARD_TOKEN) sent as a header on every request. This is a single-user tool, so that's sufficient — it's not meant to support multiple separate accounts.
  • Firestore data (tasks/logs/requests) is scoped per-uid via security rules — readable and writable only by the session that created it.
  • The secrets collection in Firestore is fully locked (allow read, write: if false) — vestigial from an earlier architecture, kept locked down since nothing should ever write there.

Known Limitations

  • No automatic detection of "this model needs your input" — Requests are manually created or seeded, not generated by an actual model signal.
  • Gemini, Mistral, and Workers AI can't show live rate-limit gauges — those providers don't expose the headers needed, full stop, not a bug to fix.
  • Meshy and Suno integration code is written but not verified against live traffic — their exact request/response shape should be double-checked against current docs before depending on it.
  • Tasks run sequentially, one at a time, even when steps don't strictly depend on each other — simple and predictable, but not the fastest possible execution.
  • "Refresh Usage" makes a real, tiny API call each time you click it — cheap, but not literally free, and not something to auto-poll.

Troubleshooting

Backend shows OFFLINE

Check both halves independently: Firebase config filled in correctly (not still the PASTE_YOUR_... placeholders), and WORKER_URL/DASHBOARD_TOKEN matching exactly what you deployed. A hard refresh (Ctrl/Cmd+Shift+R) rules out a stale cached copy.

"Unauthorized" when saving a key

The token in the dashboard doesn't match what's set on the Worker. Test directly against the Worker with curl to isolate which side is wrong before changing anything.

A model call fails immediately

Almost always account-side, not code-side — check the provider's billing/credits page. A 400 usually means a bad request (wrong model name); a 429 usually means quota or billing, not "too many requests" in the literal sense.

Firestore permission-denied

Confirm firestore.rules is actually deployed (firebase deploy --only firestore:rules) and that you're signed in (check console.log for "Signed in: ..." — anonymous auth needs to succeed before any read/write will pass the rules.)
